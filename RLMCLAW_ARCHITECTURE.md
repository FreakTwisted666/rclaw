# RLMClaw Architecture - A Recursive Language Model Agent

## 1. Vision & Goals

**Vision:** To create an autonomous agent powered by Recursive Language Models (RLM) capable of sophisticated task decomposition, execution in sandboxed environments, and robust long-term memory, serving as a highly effective personal assistant. This agent will leverage the power of RLM's self-recursive capabilities to handle complex problems and provide deep insights, while integrating with common communication channels and tools.

**Goals:**
*   Implement a core RLM inference engine for task decomposition and execution.
*   Integrate a flexible and secure sandboxing mechanism for code execution.
*   Provide robust long-term memory management and context handling.
*   Enable seamless communication through various channels (Telegram, Discord, WhatsApp).
*   Offer a comprehensive toolset for web interaction, scheduling, and system operations.
*   Maintain full trajectory logging and provide visualization for debugging and analysis.
*   Achieve high autonomy in problem-solving and self-improvement.

## 2. Core Components

### 2.1 RLM Inference Engine (rlm/engine.py)
*   **Purpose:** The brain of RLMClaw, responsible for recursive task decomposition, sub-call management, and contextual reasoning.
*   **Key Functionality:**
    *   `rlm.completion(prompt, model, environment, logger)`: Replaces standard LLM completion with recursive calls.
    *   **OQ-01: Constrained Decoding & Schema Validation:** Support for passing JSON schemas to enforce structured output from the model. Includes validation against Zod/Pydantic schemas and a 3-retry fallback mechanism with error feedback, defaulting to single-shot if retries fail.
    *   Manages a REPL-like environment for the LM to interact with.
    *   Handles variable context passing between recursive calls.
*   **Dependencies:** `rlm` library.

### 2.2 Prompt Sanitization Layer (rlm/prompt_sanitizer.py)
*   **Purpose:** A mandatory intercept point to clean and secure inbound prompts before they enter the RLM recursion engine.
*   **Rules Enforced (RISK-02):**
    1.  **Instruction injection stripping:** Removes patterns like `Ignore previous instructions`, `You are now`, `System:`, `[INST]`, `<|system|>`.
    2.  **Role override blocking:** Detects and rejects prompts attempting to redefine the model's role or persona.
    3.  **Template slot injection:** Wraps user content interpolated into meta-prompt templates with `<user_input>` delimiters.
    4.  **Length cap:** Hard limit on user-supplied prompt length (configurable, default 8000 tokens).
    5.  **Audit logging:** Logs every sanitized/blocked input with a reason code.
*   **Integration:** Fires on every inbound request before any other module sees the prompt (P5 gate before P6 API layer).

### 2.3 Environments/Sandboxes (rlm/environments/)
*   **Purpose:** Provide secure and isolated execution contexts for code generated by the RLM.
*   **Types:**
    *   `local.py`: Basic execution within the agent's Python environment (for low-risk tasks).
    *   `docker.py`: Executes code within a Docker container (default for safer execution).
    *   `modal.py`, `prime.py`, `e2b.py`: Integrations for cloud-based sandboxes (requires API keys).
*   **Interface:** All environments will expose a common interface for `execute_code(code: str, timeout: int) -> ExecutionResult`.

### 2.4 External Evaluator (rlm/evaluator.py)
*   **Purpose:** Provide an independent, LLM-as-judge mechanism for objective scoring and quality control of RLM outputs.
*   **Key Functionality (OQ-02):**
    *   A lightweight sub-module under a `TerminationController` (to be defined later).
    *   Makes a separate OpenClaw inference call with a strict scoring prompt.
    *   Returns `{ score: number, dimensions: string[], reasoning: string }`.
    *   Replaces self-scoring; self-scoring remains optional for dev/testing.
*   **Configuration:** `evaluator.enabled: true`, `evaluator.rubric: "code" | "document" | "logic" | "custom"`.

### 2.5 Context Management (rlm/context_manager.py)
*   **Purpose:** Manages the active context window for the RLM, including conversation history and long-term memory snippets, using a sliding window approach with incremental summarization (OQ-05).
*   **Strategy (OQ-05 Details):**
    1.  **Verbatim Iterations:** Keeps the last `N` iterations verbatim in the active context window (configurable, default 3).
    2.  **Incremental Summarization:** Iterations older than `N` are passed through a summarization sub-call and compressed to 2-3 sentences each.
    3.  **Summary Storage:** Summaries are stored in the State Manager (e.g., Redis) as `iteration.summary` alongside the full text.
    4.  **Context Window Assembly:** Assembled as: `[COMPRESSED_HISTORY] + [LAST_N_FULL_ITERATIONS] + [CURRENT_PROMPT]`.
*   **Configuration:** Will integrate with LLM context window size settings, with a recommended minimum of 24576 tokens to allow sufficient room before compression.

### 2.6 State Management (rlm/memory/session_manager.py)
*   **Purpose:** Stores and retrieves session-specific state, conversation history, and incremental summaries efficiently.
*   **Backend (OQ-04):** Recommended `Upstash Redis` for cloud-based session state.
    *   Serverless, pay-per-request, REST API, built-in TTL.
*   **Configuration:** `rlm.stateBackend: "redis"`, `rlm.redis.url: <Upstash URL>`, `rlm.redis.token: <token>`. TTL maps directly to Redis key expiry.

### 2.7 Tool Integrations (tools/)
*   **Purpose:** Provide RLM with access to external capabilities and information. Each tool will be a Python module.
*   **Common Tool Interface:** Each tool will have functions that the RLM can call, designed to be atomic and well-documented for the LM.
*   **Initial Tools:**
    *   `telegram.py`: Send messages, reactions, manage groups.
    *   `exec.py`: Execute shell commands (sandboxed via environment).
    *   `cron.py`: Schedule/manage periodic tasks.
    *   `memory.py`: Read/write/search long-term and short-term memory files.
    *   `web_search.py`: Perform web searches (e.g., using Brave Search API or DuckDuckGo).
    *   `files.py`: Basic file operations (read, write, list, delete) within the workspace.

### 2.8 Workspace Management (workspace/)
*   **Purpose:** Store persistent data, configuration, and agent state.
*   **Structure:**
    *   `memory/`: Long-term memory (`MEMORY.md`) and daily notes (`YYYY-MM-DD.md`).
    *   `sessions/`: Conversation history for active sessions (managed by State Management).
    *   `skills/`: User-defined skills (Python modules for RLM).
    *   `config.json`: Main configuration file.
    *   `HEARTBEAT.md`: Periodic tasks for autonomous execution.
    *   `IDENTITY.md`, `SOUL.md`, `USER.md`, `TOOLS.md`: Agent personality and operational guidelines.

### 2.9 Trajectory Logging & Visualization (rlm/logger.py, visualizer/)
*   **Purpose:** Capture the full execution path of RLM calls for analysis, debugging, and understanding agent behavior.
*   **Logging:** JSONL format, compatible with RLM's visualizer.
*   **Visualizer:** A Node.js-based web interface to explore logged trajectories.

## 3. Workflow & Execution Flow

1.  **Incoming Request:** User message arrives via a channel (e.g., Telegram).
2.  **Prompt Sanitization (RISK-02):** The `PromptSanitizer` processes the raw user input, stripping injections, blocking role overrides, wrapping template slots, and enforcing length caps. If blocked, an audit log is created.
3.  **Context Assembly:** The `ContextManager` assembles the active context window, combining compressed history, last N verbatim iterations, and the sanitized current prompt.
4.  **RLM Execution:** The primary `rlm.completion()` call is made with the sanitized prompt and assembled context.
    *   The RLM engine uses its recursive nature to break down the task, enforcing **OQ-01 (Constrained Decoding)** for structured outputs.
    *   It may decide to call various tools (e.g., `web_search`, `files`, `exec`) via its sandboxed environment.
    *   It generates code snippets or sub-prompts for further recursive calls.
    *   **OQ-03 (Concurrent Requests):** RLMClaw's design will allow for concurrent handling of recursive sub-tasks, respecting OpenClaw's Gateway scheduling queue and `OLLAMA_NUM_PARALLEL` settings where applicable.
5.  **Tool Execution:** When a tool is called, the RLM generates Python code, which is then passed to the configured environment (e.g., `docker.py`) for execution.
6.  **Result Integration & Evaluation:** Tool outputs are fed back into the RLM's context. The `ExternalEvaluator` (OQ-02) may be invoked to score RLM outputs before final integration.
7.  **State Update:** The `StateManager` updates session history and potentially stores incremental summaries (OQ-05).
8.  **Final Response:** Once the RLM determines the task is complete, it formulates a final response.
9.  **Post-processing & Output:** Response is sent back to the user via the appropriate channel. Relevant context is updated in `sessions/` and `memory/`.

## 5. Setup & Development Flow

1.  **Clone Repository:** `git clone https://github.com/FreakTwisted666/rclaw.git`
2.  **Install Dependencies:** `uv venv && source .venv/bin/activate && uv pip install -e .` (within rlmclaw dir)
3.  **Configure:** Edit `config.json` with API keys and preferences.
4.  **Run:** `./main.py` or equivalent entry point.
